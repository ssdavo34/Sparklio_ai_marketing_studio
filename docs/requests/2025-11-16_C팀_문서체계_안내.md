## 📋 공문 요약 (1분)

## 🎯 왜 문서 체계가 필요한가?

### 현재 문제점

**Before (체계 없음)**:
```
개발자: "006번 방식이 뭐예요?"
리더: "음... Obsidian 어디쯤 있을 텐데..."
개발자: "경로가 어떻게 되나요?"
리더: "K:\obsidian-k\... (5분 설명)"
개발자: "제 PC에는 obsidian-k 폴더가 없는데요?"
```

**After (체계 있음)**:
```
개발자: "Gateway Pattern 문서 어디 있나요?"
리더: "00_INDEX.md → ARCH-002"
개발자: (10초 후) "찾았습니다!"
```

### C팀이 겪을 수 있는 문제

1. **컴포넌트 설계 문서를 찾을 수 없음**
   - "Canvas Studio 컴포넌트 구조가 어디 있지?"
   - 개인 메모? 슬랙? Git 커밋 메시지?

2. **API 스펙 불일치**
   - Backend API 변경 → Frontend 모름
   - 문서 업데이트 → 어디에?

3. **신규 팀원 온보딩**
   - "Canvas Studio 어떻게 작동해요?"
   - 매번 설명 반복

## 📝 문서 작성 규칙 (C팀용)

### 1. 문서 ID 부여

**모든 공식 문서에는 고유 ID 부여**:

```yaml
---
doc_id: ARCH-004              # 카테고리-번호
title: Canvas Studio 아키텍처
created: 2025-11-17
updated: 2025-11-17
status: draft                 # draft | review | approved
authors: C팀 (Frontend)
related:
  - ARCH-001                  # 관련 문서 ID
  - SPEC-003
---
```

**C팀이 사용할 문서 ID 접두사**:
- `ARCH-00X`: 아키텍처 (Canvas Studio 구조 등)
- `SPEC-00X`: 기술 스펙 (Component Spec 등)
- `DEC-00X`: 의사결정 (왜 React?, 왜 Fabric.js?)
- `GUIDE-00X`: 개발 가이드
- `REPORT-00X`: 분석 보고서

### 2. 파일 명명 규칙

**날짜 기반**:
```
YYYY-MM-DD_일련번호_제목.md
예: 2025-11-17_001_CANVAS_COMPONENT_DESIGN.md
```

**버전 기반**:
```
제목_vX.Y.md
예: COMPONENT_SPEC_v1.0.md
```

**현재 상태**:
```
CURRENT_*.md
예: CURRENT_C팀_SPRINT.md
```

### 3. 문서 헤더 필수 항목

```yaml
---
doc_id: ARCH-004
title: 문서 제목 (한글 OK)
created: 2025-11-17
updated: 2025-11-17
status: draft                 # draft | review | approved | deprecated
authors: C팀 (홍길동, 김철수)
reviewers: PM, A팀           # 검토자
related:                      # 관련 문서
  - ARCH-001
  - SPEC-003
supersedes: null              # 이 문서가 대체한 이전 문서
---
```

### 4. TL;DR 섹션 필수

**모든 문서는 30초 요약 포함**:

```markdown
## TL;DR (30초 요약)

- 핵심 내용 1 (한 줄)
- 핵심 내용 2 (한 줄)
- 핵심 내용 3 (한 줄)

**결론**: 한 문장 요약
```

## 🔗 문서 참조 방법

### ❌ 나쁜 예

```markdown
006번 방식으로 진행합니다.
Canvas 컴포넌트는 저번에 만든 그거 쓰세요.
홍길동님이 작성한 문서 참고하세요.
```

**문제점**:
- "006번"이 뭔지 모름
- "저번에 만든 그거"를 찾을 수 없음
- "홍길동님 문서"가 여러 개

### ✅ 좋은 예

```markdown
Gateway Pattern 채택 (참조: [ARCH-002](../architecture/002_GATEWAY_PATTERN.md))

Canvas 컴포넌트 구조 참조: [ARCH-004: Canvas Studio Architecture](../architecture/004_CANVAS_STUDIO.md)

API Contract 스펙: [SPEC-001](../specs/API_CONTRACT_v1.0.md)
```

**장점**:
- 클릭 한 번에 이동
- 문서 ID로 검색 가능
- 누가 작성했든 찾을 수 있음

```yaml
  - ARCH-001: System Overview
```

```markdown
# Canvas Studio Architecture

...
```

## ❓ FAQ

### Q1: 기존 슬랙/노션 문서는 어떻게 하나요?

**A**:
- **슬랙/노션**: 임시 논의/브레인스토밍 용도로 계속 사용
- **공식 문서**: 확정된 내용만 `docs/` 폴더로 이관
- **규칙**: "1주일 지나도 참조되는 내용"은 공식 문서화

### Q2: 문서 작성 시간이 부족해요

**A**:
- **최소 요구사항**: TL;DR + 핵심 섹션만 (10분)
- **상세 내용**: 나중에 추가 가능 (status: draft)
- **협업**: 팀원과 분담 작성

### Q3: 문서 ID를 어떻게 정하나요?

**A**:
- **00_INDEX.md 확인**: 마지막 번호 + 1
- **예시**: ARCH-003까지 있으면 → ARCH-004 사용
- **충돌 시**: A팀에게 문의

### Q4: 문서 리뷰는 누가 하나요?

**A**:
- **작성자**: C팀 (Frontend)
- **1차 리뷰**: C팀 내부 (동료 리뷰)
- **2차 리뷰**: PM 또는 A팀 (선택적)
- **승인 후**: status: approved

### Q5: 영어로 써야 하나요?

**A**:
- **문서 내용**: 한글 OK (편한 언어로)
- **코드/파일명**: 영어 권장
- **문서 ID**: 영어 (ARCH, SPEC 등)
